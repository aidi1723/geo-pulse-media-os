# GEO-Pulse Maintenance Guide

This guide records where to maintain common product behavior, UI content, mock API data, tests, and delivery notes.

## Maintenance Scope

GEO-Pulse is currently a React + Vite prototype with a local Node mock API. Treat it as a demo-ready MVP shell, not a production backend.

Primary maintenance areas:

- Frontend workspace and UI flow: `src/`
- Local API and state persistence: `server/`
- Regression coverage: `tests/`
- Product and architecture documentation: `README.md`, `CHANGELOG.md`, `DESIGN.md`, `docs/`
- Built preview artifacts: `dist/`

## Common Update Paths

### Dashboard Shell and Page Routing

Path: `src/App.jsx`

Use this file for:

- App-level layout composition
- Sidebar active view routing
- Bootstrap flow from local API
- Scenario switching
- Job detail loading
- Returning from a job artifact to the right workspace section

Do not add large data-mapping or API mutation logic here. Put reusable workspace state rules in `src/state/`, controller logic in `src/hooks/`, and async workflow mutations in `src/actions/`.

### Workspace State

Paths:

- `src/state/workspaceState.js`
- `src/hooks/useWorkspaceController.js`

Use `src/state/workspaceState.js` for pure state mapping:

- Initial workspace state
- Bootstrap payload mapping
- Fallback values when the API omits optional app data

Use `src/hooks/useWorkspaceController.js` for React state ownership:

- Workspace state storage
- Focused setters such as selected topic, banner, jobs, and selected job id
- Applying bootstrap payloads through the pure state mapper

When adding a new workspace field, update both:

- `createInitialWorkspaceState`
- `applyWorkspacePayloadToState`

Then add or update `tests/src/workspace-state.test.mjs` and `tests/ui/workspace-controller.test.jsx`.

### Workflow Actions

Path: `src/actions/workflowActions.js`

Use this file for async workflow actions that combine an API call with workspace/UI updates:

- Generate draft
- Run daily workflow
- Refresh topics
- Schedule distribution

These actions receive dependencies through parameters. Keep that pattern so behavior can be tested without real network calls.

When adding a new workflow action:

1. Add a behavior test in `tests/src/workflow-actions.test.mjs`.
2. Add the action to `createWorkflowActions`.
3. Inject the required service in `src/App.jsx`.
4. Wire the action to the relevant section component.

### UI Sections

Paths:

- `src/sections/OverviewSection.jsx`
- `src/sections/DiscoverySection.jsx`
- `src/sections/StudioSection.jsx`
- `src/sections/VideoSection.jsx`
- `src/sections/DistributionSection.jsx`
- `src/sections/SecuritySection.jsx`

Use sections for page-level UI only. They should receive data and callbacks through props.

Do not import `src/data/mockData.js` directly from sections when the value can come from bootstrap payloads. This keeps the UI ready for real API data.

### Shared Components

Path: `src/components/`

Use shared components for repeated UI elements:

- `TaskBoard.jsx`
- `JobColumn.jsx`
- `JobDetailCard.jsx`
- `MetricCard.jsx`
- `Sidebar.jsx`
- `StatusBadge.jsx`

Before adding a new repeated UI pattern, check whether it belongs here instead of a section file.

### Styling and Design Rules

Paths:

- `DESIGN.md`
- `src/styles.css`

`DESIGN.md` is the visual source of truth. For UI polish or design-system work, update shared tokens and component classes in `src/styles.css` before adding page-local styling.

Maintain these rules:

- Use semantic CSS variables for colors and surfaces.
- Preserve visible focus states.
- Keep loading and disabled states explicit.
- Check responsive behavior after layout changes.

### Mock Data and Local API

Paths:

- `src/data/mockData.js`
- `server/domain.mjs`
- `server/router.mjs`
- `server/state-store.mjs`
- `server/data/state.json`

Use `src/data/mockData.js` for demo seed content and fallback static data.

Use `server/domain.mjs` for local API domain behavior:

- Draft composition
- Job creation
- Job action rules
- Bootstrap payload construction

Use `server/router.mjs` for HTTP endpoint routing only.

Use `server/state-store.mjs` for JSON persistence. It supports `createStateStore(stateFile)` for isolated tests and default `readState` / `updateState` for the local API.

Reset local demo data with:

```bash
npm run reset:api-state
```

### Tests

Paths:

- `tests/server/`
- `tests/src/`
- `tests/ui/`

Test placement:

- Server/API behavior: `tests/server/*.test.mjs`
- Pure utilities and action factories: `tests/src/*.test.mjs`
- React component and hook behavior: `tests/ui/*.test.jsx`

Run all tests with:

```bash
npm test
```

Add new tests to the `test` script in `package.json` so they run by default.

### Build Artifacts

Path: `dist/`

`dist/` is regenerated by:

```bash
npm run build
```

Only treat `dist/` as deployable output after `npm test` and `npm run build` both pass.

## Content Update Checklist

When updating product content or demo behavior:

1. Update mock seed data in `src/data/mockData.js` or API domain behavior in `server/domain.mjs`.
2. If the content is returned through bootstrap, verify `src/state/workspaceState.js` maps it.
3. If UI controls display it, ensure the section receives it through props.
4. Add or update tests in `tests/src/`, `tests/server/`, or `tests/ui/`.
5. Update `README.md` when user-facing setup, endpoints, or structure changes.
6. Update `CHANGELOG.md` with the date, changed files, behavior impact, and verification commands.
7. Run `npm test`.
8. Run `npm run build`.

## GitHub Release Checklist

Before pushing to GitHub:

1. Confirm the project is inside a git repository.
2. Confirm the remote with `git remote -v`.
3. Review changed source and generated files.
4. Run `npm test`.
5. Run `npm run build`.
6. Commit with a clear message.
7. Push to the intended branch.
